# Milestones, Not Release Churn

Grandview reached a sound result through a long chain of dependent evidence: source fitness, official-reference inspection, background evidence, proposal, owner surface, and response intake. Each step kept excellent provenance, but each also repeated the full issue, release, tag, and lifecycle machinery. The work was safe; the cadence was expensive.

BL-GOV-003 separates the useful atomicity from the release overhead. An evidence unit still gets its own immutable identity, inputs, hashes, gates, disposition, and retained failures. Related units can now accumulate inside one milestone issue and branch. A milestone ships only when the batch changes accepted project truth or supports a material remediation, fallback, deferral, or stop decision.

The policy keeps an escape hatch for real risk. Licensing, security, custody, rollback, stop-condition, and urgent-correctness defects may still ship immediately as exception checkpoints. Convenience and small diff size do not qualify.

The first use is Petes Lake. BL-GOV-003 designates issue #521 for post-merge revision into one six-event evidence milestone containing its complete source-to-owner-response chain. Even a successful sixth event will not manufacture a dataset: BurnLens must still pass the separate dataset-fitness milestone before any dataset, split, baseline, or model work can begin.

This governance change modifies no software, evidence byte, owner decision, label, data, model, deployment, tag, or scientific gate. It changes how often BurnLens pays the release cost while keeping every evidence gate visible.
