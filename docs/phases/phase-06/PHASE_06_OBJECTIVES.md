# BurnLens Deschutes — Phase Six Objectives

## Document role

This document expands the Phase Six objective summarized in `docs/roadmap/BURNLENS_BUILD_ROADMAP.md`.

It defines the final publication and closeout outcome. It does not authorize production promotion, tags, GitHub Releases, public downloads, or licensing decisions before their explicit gates.

## Phase name

**Portfolio Publication, Public Release, Technical Communication, and Project Closeout**

## Canonical objective summary

Publish one coherent, traceable portfolio release whose demo, case study, repository, presentation, licensing, citation, archive, and maintenance posture all refer to the same accepted evidence and limitations.

## Current status

**Proposed — blocked.**

Phase Six depends on an accepted Phase Five release candidate, verified public claims, resolved licensing for every published artifact, exact in-repository version linkage, and a reversible verified release path.

## Phase purpose

Phase Six converts the accepted release candidate into a public body of evidence that a technical reviewer can understand, inspect, reproduce, evaluate, cite, and responsibly interpret.

The final portfolio should demonstrate:

1. technical capability;
2. reliability and reversible release control;
3. bounded planning-style usefulness;
4. transparent methods and provenance;
5. strong communication and visual evidence;
6. responsible limitations and prohibited uses;
7. prompt-assisted development without exposing private reasoning or credentials.

The final release may be model-centered, model-with-caveats, baseline-first, or a technical case study without interactive output. Evidence, not ambition, selects the outcome.

## Position in the six-phase build

| Phase Five provides | Phase Six creates | Final result |
|---|---|---|
| Frozen candidate, canonical run, QA/security/accessibility evidence, version audit, known issues, rollback target, approved claims | Production deployment, case study, public documentation, repository releases, citation/licensing records, presentation package, archive, maintenance and closeout record | One coherent, traceable portfolio release or documented narrower/withdrawn outcome |

## Required outcomes

Phase Six must produce:

1. one frozen public-release scope and canonical demonstration run;
2. one evidence-backed portfolio case study;
3. complete public technical documentation across repository roles;
4. explicit code, documentation, model, data, asset, and derived-output licensing posture;
5. citation metadata and reproducibility snapshot;
6. one production deployment tied to the accepted candidate;
7. one or more exact GitHub release records only where separately authorized;
8. consistent metadata, social preview, sitemap, and crawler policy;
9. 30-second, 2-minute, and 5-minute narratives;
10. a fixed demonstration sequence, presentation deck, and visual evidence package;
11. preserved release archive or explicit preservation decision;
12. maintenance, issue-intake, supersession, monitoring, and closeout posture.

## Objective set

### Objective One — Authorize and freeze public scope

**Purpose:** Enumerate exactly what may be published and prevent last-minute feature or analytical expansion.

**Required result:** Accepted candidate, canonical run, exact commits and versions, artifact inventory and classifications, permitted downloads, withheld artifacts, approved claims, warnings, known issues, rollback target, and publication states.

**Acceptance gate:** No ambiguity remains about the public evidence, prohibited artifacts, licensing blockers, or release-blocking conditions.

### Objective Two — Build the case study and evidence narrative

**Purpose:** Explain the project as a disciplined engineering and analytical process rather than a collection of attractive screenshots.

**Required result:** Problem, scope, data, labels, baseline, model, evaluation, GEOINT integration, reliability, canonical result, limitations, prompt-assisted process, lessons, and next steps; deliberate visual evidence with captions, versions, sources, caveats, and alt text.

**Acceptance gate:** Every quantitative claim traces to accepted evidence; model and baseline are represented honestly; failure cases are visible; no external adoption or validation is implied.

### Objective Three — Finalize public documentation and repository experience

**Purpose:** Make this repository's analytical system, application, and case study understandable, reproducible, licensed, citable, and navigable without prior project context.

**Required result:** Clear repository roles; README and quickstart; architecture, data, labels, model, evaluation, methods, limitations, reproducibility, versioning, security, known issues, citation, licensing, run manifest, and release notes; contribution and maintenance posture.

**Acceptance gate:** A reviewer can locate methods, evidence, limitations, setup, versions, terms, and citation without searching archival history.

### Objective Four — Publish the production site and versioned releases

**Purpose:** Promote the accepted candidate to the canonical domain and create exact repository release records.

**Required result:** Repository-owned production site; public evidence sections; metadata; social preview; sitemap and crawler policy; draft-then-publish release sequence; permitted assets and checksums; exact commit/version linkage; post-deployment verification and rollback record.

**Acceptance gate:** Production displays the accepted run and version chain; releases point to exact commits; restricted assets are excluded; previews and stale deployments cannot impersonate the canonical release.

### Objective Five — Create presentation and demonstration materials

**Purpose:** Provide consistent ways to explain BurnLens to technical reviewers, hiring managers, collaborators, and general visitors without changing the evidence.

**Required result:** 30-second, 2-minute, and 5-minute narratives; fixed live demo flow; compact deck; consistent visuals and metrics; screenshot protocol; optional captioned recording; precise resume language.

**Acceptance gate:** Every communication surface uses the same project name, AOI, versions, run, metrics, dates, warnings, and evidence classifications.

### Objective Six — Preserve, maintain, and close the build

**Purpose:** Preserve the accepted release, define future support and supersession, and record the outcome of every phase.

**Required result:** Archive or DOI decision; active/limited/frozen/archived maintenance posture; issue-intake policy; post-release monitoring; public metrics with caveats; supersession rule; six-phase closeout and final outcome.

**Acceptance gate:** The project has a durable archive, clear maintenance expectations, exact supersession history, and one truthful final status.

## Public artifact classes

| Class | Examples |
|---|---|
| Source/reference | Public imagery metadata, official planning or fire references |
| Dataset or label evidence | Dataset manifest, label schema, split summary |
| Baseline evidence | Threshold or reference-derived baseline outputs |
| Model evidence | Metrics, predictions, model card, failure examples |
| Integrated run evidence | Rasters, vectors, overlays, summary, manifest |
| Reliability evidence | Tests, QA, security, accessibility, reproducibility |
| Portfolio interpretation | Case-study narrative, diagrams, captions, deck |
| Release metadata | Version, tag, checksums, release notes, citation |

These classes must remain distinguishable enough that interpretation is not mistaken for source, label, baseline, or model evidence.

## Completion evidence

Phase Six is complete only when:

- one canonical run is used consistently across the repository-owned site, screenshots, case study, deck, and release notes;
- all public versions and commits reconcile;
- every public artifact has an explicit license/terms and attribution decision;
- the production deployment passes post-release verification;
- published assets match recorded checksums;
- methods, evaluation, limitations, use boundaries, and source precedence are easy to find;
- release, citation, archive, rollback, and supersession records are preserved;
- presentation claims match the accepted evidence;
- the maintenance and issue-intake posture is public;
- the six-phase closeout selects one truthful final outcome.

## Dependencies

- accepted Phase Five release candidate and rollback target;
- approved public claims and warning language;
- resolved source, code, documentation, asset, model-weight, and derived-artifact licensing;
- canonical domain and deployment authorization;
- exact analytical and application commits from this repository;
- authorized tag and GitHub Release actions where applicable.

## Non-goals

Phase Six does not:

- retrain, retune, change thresholds, add data, change AOI, or regenerate analytical results without returning to the controlling phase;
- add live services, user uploads, incident feeds, or a second CV task;
- combine outputs from different runs into a fictional single result;
- imply agency request, practitioner validation, operational readiness, field validation, risk reduction, or adoption;
- publish source-restricted imagery or derived artifacts merely for completeness;
- expose private prompts, chain-of-thought, secrets, credentials, or raw private session material.

## Fixed boundaries

- One canonical public run.
- Public narrative never overrides contradictory technical evidence.
- The portfolio is experimental and non-operational.
- Official sources govern.
- Limitations and failure cases are central, not hidden in a footer.
- A public repository is not assumed open source without an explicit license.
- A deployment is not called production-ready wildfire infrastructure.
- Tags and GitHub Releases require issue-backed evidence and verification under the controlling goal.

## Known risks and assumptions

- Stale screenshots, metrics, or copy can diverge from the canonical run.
- One license may not cover code, documentation, weights, imagery, overlays, basemaps, fonts, and derived artifacts.
- Social preview imagery can overstate the project if it is more dramatic or less caveated than the live evidence.
- Search indexing may expose preview or fixture routes unless crawler controls are deliberate.
- Recorded demos may become stale and require a supersession notice.
- Portfolio language may drift toward impact or adoption claims unsupported by internal evidence.
- Public downloads may expose restricted data, local paths, or sensitive metadata unless audited.

## Authority delegated to Codex

After the predecessor evidence gate and within the frozen release scope, Codex may:

- draft and refine the case study, documentation, release notes, metadata, presentation, captions, and closeout records;
- correct broken links, citations, accessibility defects, version inconsistencies, misleading wording, and release-blocking documentation defects;
- assemble permitted assets and checksums;
- deploy repository-owned previews and production checkpoints and perform post-deployment verification;
- recommend model-centered, caveated, baseline-first, case-study-only, maintenance, archive, or withdrawal outcomes.

Codex must not silently change analytical evidence or the canonical run, proceed with unresolved licensing, change access/ownership/public-sharing status without owner approval, or ship an unverified release.

## Changes requiring owner approval

The stop conditions in `docs/governance/BURNLENS_EXECUTION_GOAL.md` control. Codex may freeze the evidence-backed public scope, select the canonical verified run, create versions/tags/releases, deploy from this repository, and choose the next reversible release or maintenance checkpoint without routine approval.

Owner approval is required before changing the core promise, target user, CV task, phase outcomes, or use boundaries; crossing a no-go boundary; proceeding with unresolved licensing/terms; spending money or adding a paid service/secret; changing access, ownership, public-sharing status, or the canonical domain; taking an irreversible action; implying official/operational/emergency-ready/field-validated/endorsed status; or shipping something unverifiable.

## Final outcome options

- **Published — model-centered portfolio release**
- **Published — model with caveats**
- **Published — baseline-first portfolio release**
- **Published — technical case study without interactive output**
- **Maintenance mode**
- **Archived**
- **Withdrawn**

The evidence and release gate select the outcome. A narrower, honest release is preferable to an unsupported broader one.

## Expected closeout record

The final closeout must state for each phase:

- intended objective;
- actual outcome;
- accepted versions and artifacts;
- major evidence;
- carried limitations;
- deviations from the roadmap and their rationale;
- unresolved issues;
- final public claims permitted and prohibited;
- maintenance or supersession path.
