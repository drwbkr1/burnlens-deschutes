# P1O2-T04 — CV Output Contract

## Status

Phase 1 / Objective Two task artifact.

This document defines the expected output contract for future BurnLens Deschutes segmentation or baseline outputs. It does not create model predictions, rasters, vectors, maps, reports, or UI layers.

## Applies to

Current Objective Two decisions:

- CV task: experimental binary semantic segmentation for wildfire-relevant screening
- Primary target: active-fire / hotspot-informed binary fire mask
- Fallback target: burn-scar binary mask if active-fire label feasibility fails
- Class handling: positive, negative/background, and unknown/exclude/review-needed

## Required workflow fit

Every future CV or baseline output must fit this locked BurnLens workflow chain:

```text
imagery → preprocessing → segmentation or baseline mask → raster output → vector polygons → map overlay → exposure-style summary → documented run package
```

The output contract is designed to make sure the future model is not an isolated notebook artifact. It must produce or support geospatial outputs that can be traced, inspected, documented, and reused in a portfolio case study.

## Output levels

| Level | Output | Required in first model phase? | Purpose |
|---|---|---:|---|
| 0 | Input tile or scene reference | Yes | Connects output back to source imagery and preprocessing metadata. |
| 1 | Binary mask | Yes | Core segmentation output: fire-relevant vs. background. |
| 2 | Unknown/exclude mask | Yes, if ambiguous regions exist | Preserves ambiguous or unusable pixels/regions without forcing them into binary labels. |
| 3 | Probability or confidence raster | Preferred when model architecture supports it | Helps distinguish thresholded output from uncertainty-aware interpretation. |
| 4 | Georeferenced raster | Required for GEOINT integration phase | Converts the mask into a spatial artifact with CRS, transform, extent, and timestamp. |
| 5 | Vector polygons | Later integration phase | Supports map overlays, summaries, and inspection of detected areas. |
| 6 | Exposure-style summary fields | Later integration phase | Supports planning-style summaries such as nearby roads, structures, or reference layer overlap. |
| 7 | Run package | Required once inference exists | Records provenance, versions, inputs, warnings, and output paths. |

## Minimum binary mask contract

A future binary mask must include:

- same tile dimensions as the prepared input or a documented resize mapping
- explicit positive value
- explicit negative/background value
- no silent conversion of ambiguous regions into either class
- source image identifier
- preprocessing version
- label schema version, when labels are involved
- model or baseline version, when generated
- run ID, when generated through an inference or baseline run

Recommended logical values:

| Value | Meaning |
|---:|---|
| 1 | Positive / fire-relevant |
| 0 | Negative / background |
| 255 or null | Unknown / exclude / review needed |

## Minimum geospatial raster contract

When masks become geospatial rasters in a later phase, each raster should carry or be accompanied by:

- CRS / projection
- affine transform or geotransform
- bounds / extent
- pixel size / resolution
- nodata value
- source imagery ID
- source imagery date/time if available
- processing timestamp
- model or baseline version
- run ID
- warnings or degradation flags

## Vector polygon contract

When future masks are converted into polygons, each feature should include at minimum:

| Field | Meaning |
|---|---|
| `run_id` | Unique run that produced the feature. |
| `output_type` | `model`, `baseline`, or `reference`. |
| `class_value` | Positive/background/exclude class value when applicable. |
| `class_name` | Human-readable class name. |
| `model_version` | Model version, if model-generated. |
| `baseline_version` | Baseline version, if baseline-generated. |
| `source_image_id` | Source image or scene reference. |
| `source_datetime` | Source acquisition time/date, if available. |
| `processed_at` | Processing timestamp. |
| `area_m2` | Polygon area in square meters, when safe to compute. |
| `confidence_mean` | Mean confidence/probability, if available. |
| `warning_flags` | Cloud, smoke, stale source, alignment issue, low confidence, or similar. |
| `use_boundary` | Compact note that the output is experimental and non-operational. |

## Output type separation

BurnLens must always separate these three output categories:

| Category | Description | Allowed claim |
|---|---|---|
| `model` | Output generated by a trained BurnLens model. | Experimental model-generated segmentation output. |
| `baseline` | Output generated by public reference data, thresholding, buffering, or other non-ML logic. | Experimental baseline or reference-derived screening layer. |
| `reference` | Official/public source layer used for context or comparison. | Source layer from named public/official provider; official source terms govern. |

A model output must never be presented as an official source layer. A baseline output must never be presented as ground truth. A reference layer must never be silently modified without provenance.

## Summary-ready metadata

Future summaries should be derived only after spatial output exists. Minimum summary-ready fields should include:

- run ID
- AOI ID and AOI version
- source date/time
- processing timestamp
- output type
- model or baseline version
- total positive area, if meaningful
- count of polygons or positive components, if meaningful
- overlap with reference layer, if computed
- nearest or intersecting planning overlays, if computed later
- warning flags
- use-boundary statement

## Run package linkage

Every future output must point back to a run package when runs exist.

Recommended future structure:

```text
runs/
  BL-YYYY-MM-DD-deschutes-aoiXX-mXXX-dXXX/
    run_manifest.json
    prediction_mask.tif
    prediction_confidence.tif
    prediction_polygons.geojson
    exposure_summary.json
    run_report.md
```

This task does not create that structure; it defines the contract for later phases.

## Quality and stop conditions

Future outputs should be flagged or withheld when:

- source imagery is stale for the claimed use
- cloud, smoke, shadow, snow, water glare, or other artifacts block interpretation
- CRS or georeferencing is missing or inconsistent
- input and mask dimensions do not align
- model/baseline version cannot be identified
- run ID is missing
- source metadata is incomplete
- output conflicts with an official/reference source in a way that is not documented

## Use boundary

All future BurnLens CV outputs must carry this meaning:

> Experimental BurnLens segmentation or baseline output for portfolio demonstration and planning-style screening. Not official wildfire information. Not emergency guidance. Official sources govern.
