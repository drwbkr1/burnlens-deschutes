# BurnLens Deschutes — Transparency Requirements

## Purpose

This document defines the transparency and trust requirements for BurnLens Deschutes.

BurnLens Deschutes should earn trust through source documentation, method clarity, version traceability, visible limitations, and careful claims. Reviewers should not have to trust an output simply because it was generated by AI.

## Transparency principle

Every public BurnLens Deschutes output should answer six questions:

1. What source data was used?
2. What processing step created this output?
3. What model or baseline method was used?
4. What version, commit, and run ID produced it?
5. What limitations or warnings apply?
6. What claims are not supported by this output?

If an output cannot answer those questions, it is not portfolio-ready.

## Required transparency artifacts

The Objective One document set should include:

- `PROJECT_IDENTITY.md`
- `PROJECT_THESIS.md`
- `PROJECT_CATEGORY.md`
- `TARGET_USER.md`
- `PRIMARY_USER_STORY.md`
- `AOI_RATIONALE.md`
- `DATA_PREMISE.md`
- `TECHNICAL_DESCRIPTION.md`
- `USE_BOUNDARIES.md`
- `SOURCE_PRECEDENCE.md`
- `TRANSPARENCY_REQUIREMENTS.md`
- `MODEL_CARD_TEMPLATE.md`

Later phases should add `DATASET_CARD_TEMPLATE.md`, `RUN_MANIFEST_TEMPLATE.json`, `CLAIMS_REGISTER.md`, and release documentation.

## Source transparency

Every source used in a public run should record source name, organization, URL, access method, source date, download date, license or use terms, spatial coverage, CRS, resolution or scale, intended BurnLens use, known limitations, local filename, source version, and checksum when practical.

## Model transparency

Any trained model should have a model card before it is referenced publicly. The model card should document model version, architecture, task, intended use, unsupported uses, training data version, label schema version, AOI or geographic scope, evaluation data, metrics, known failure modes, source commit, training date, and release status.

If no trained model exists yet, the project should say so. Do not imply model performance before training and evaluation.

## Dataset transparency

Any dataset used for training, validation, testing, or demo output should have a dataset card or manifest documenting source inputs, geography, time period, download method, preprocessing, labeling method, split logic, gaps, quality issues, recommended uses, prohibited uses, and maintenance status.

## Run transparency

Every public run should include a manifest recording run ID, AOI version, app version, model or baseline version, dataset version, label schema version where relevant, source data versions, GitHub commit, timestamp, output files, warnings, limitations, and source-precedence note.

No screenshot, map, report, or summary should be considered final unless it points back to a run ID.

## Evaluation transparency

Evaluation should report metric definitions, validation/test split logic, sample count, geographic and temporal scope, comparison baseline, degraded conditions, false-positive examples, false-negative examples, and interpretation limits. Avoid reporting a single performance number without context.

## Limitation transparency

Common limitations to carry forward include incomplete or stale public data, cloud/smoke/snow/haze/shadow issues, active-fire detections not being exact fire perimeters, experimental model outputs not being field verified, road presence not implying road safety, and official sources governing when information differs.

## Claim control

Before public release, BurnLens Deschutes should maintain a claims register classifying public claims as supported, supported with caveat, not yet supported, or prohibited.

Supported claims may include reproducibility, traceable outputs, experimental map-ready outputs, and transparent limitations when evidence exists.

Prohibited claims include confirmed wildfire location, safe evacuation routes, official hazard determination, field validation, agency endorsement, or replacement of official wildfire information.

## Website transparency

The website should show project description, experimental-use disclaimer, source-precedence note, methods summary, release/version snapshot, GitHub link, known limitations, sample outputs with run IDs, and model status.
